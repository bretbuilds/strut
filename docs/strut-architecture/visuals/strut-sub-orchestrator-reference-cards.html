Human's change description via $ARGUMENTS. Optional --step flag for step mode.

Three phases, sequential. Thinnest orchestrator — three dispatches with status checks between them.
Resume logic
Reads process-change-state.json — if status == "blocked", this is a resume. Skip Read Truth, go directly to run-process-change. Otherwise: new run, start from Read Truth.
Step mode
If --step in $ARGUMENTS: writes .pipeline/step-mode flag file. Every orchestrator checks for this flag after each dispatch. Per-invocation — omitting --step on resume removes the flag. Cleaned up after Update Truth completes.
Returns
Pipeline complete or blocked at a human gate
Key rule: Never does work itself. Never reads agent-level result files. Dispatches phase orchestrators and checks their returns.

Sub-orchestrators (within Phase 2)
run-spec-refinement
Sub-orchestrator · spec cyclecontext: ~3–5 cycles
Receives
classification.json + impact-scan.md + truth-repo-impact-scan-result.json
Dispatch sequence
spec-derive-intent
→
spec-write
→
spec-review
spec-review performs two phases: quality check + testability check
Modifier behavior
Trust ON: spec-derive-intent populates must_never[] from risk signals + rules. spec-write creates negative-type criteria from must_never entries, each with source tracing to intent.
Decompose ON: spec-write produces task breakdown (≤5 tasks, each independently testable) instead of single task. spec-review adds Phase 3: decomposition validation — each task checked for context-fit, eval-fit, merge-fit, dependency-fit.
Retry / loop
Spec cycle: If spec-review fails → spec-write re-runs with feedback (both review_issues and validation_issues simultaneously). Max 5 total iterations. Exhausted → return failed, escalate to human.
Fail-fast
spec-derive-intent fails → return failed immediately (no spec cycle entered)
Returns
Status to run-process-change: passed | failed
Writes (internal)
intent.jsonspec.jsonspec-review.json
Key rule: spec-review has two phases (quality + testability) — spec-write gets all feedback simultaneously so it can fix both in one revision, not sequentially across iterations.
run-implementation
Sub-orchestrator · TDD cyclecontext: ~3–5 cycles/task
Receives
Approved spec.json + classification.json (for modifier state)
Dispatch sequence (per task)
git-tool (branch)
skip if exists→
impl-write-tests
→
impl-write-code
→
run-review-chain
→
git-tool (commit)
Modifier behavior
Decompose ON: Steps 2–5 repeat per task (impl-write-tests → impl-write-code → run-review-chain → commit). Writes active-task.json before each task cycle so agents resolve the active task id. Retry budget resets per task. Context compaction between tasks. After task 1 commit → return blocked with gate: "task_1" for human validation.
Retry / loop
Review chain fail: impl-write-code re-runs with feedback from review-chain-result.json. Then run-review-chain re-runs from scratch (all reviewers, clean slate). Max 3 retries. Exhausted → return failed, escalate to human.
Fail-fast
impl-write-tests fails → return failed immediately (no TDD cycle entered). impl-write-code fails → return failed immediately (no review attempted).
Human gates
Decompose ON only: Task 1 gate after first task's commit. run-implementation returns blocked with gate: "task_1". run-process-change handles the pause. On resume, re-dispatches run-implementation with start_task arg.
Returns
Status to run-process-change: passed | failed | blocked (at task 1 gate)
Writes (internal per task)
git-branch-result.jsontests-result.jsonimpl-write-code-result.jsongit-commit-result.jsonactive-task.jsonimplementation-status.json
Key rule: Delegates review to run-review-chain — does NOT dispatch review-scope / review-criteria-eval / review-security directly. Standard path (decompose OFF) has one task — loop executes once, behavior unchanged.
run-review-chain
Sub-orchestrator · reviewcontext: ~2–3 cycles
Receives
diff + spec.json (criteria + out_of_scope) + (trust ON: .claude/rules/* + must_never from spec)
Dispatch sequence
review-scope
→
review-criteria-eval
→
review-security
(trust ON only)
Sequential, fail-fast. Each reviewer gets isolated context.
Modifier behavior
Trust ON: Dispatches review-security as third step after review-criteria-eval passes. Same fail-fast behavior applies.
Fail-fast
Any reviewer fails → chain stops immediately. Returns failed with feedback from the failed reviewer. Does not run subsequent reviewers.
Retry behavior
On retry (called again by run-implementation after impl-write-code revises): always re-runs from review-scope. Revised implementation may have introduced new scope issues even if scope passed prior. All stale reviewer files removed — every execution starts clean. Max 3 total retries — budget managed by parent run-implementation.
Returns
Status to run-implementation: passed | failed (with aggregated feedback)
Writes (internal)
review-scope.jsonreview-criteria-eval.jsonreview-security.jsonreview-chain-result.json
Key rule: Each reviewer gets isolated context — diff + its specific reference (spec for review-scope, criteria for review-criteria-eval, rules for review-security). No reviewer sees another's assessment.
run-build-check
Sub-orchestrator · build verificationcontext: ~1–4 cycles
Receives
Branch with committed implementation (complete codebase on feature branch)
Dispatch sequence
build-check
(bash script)→ on fail →
build-error-cleanup
→
re-run build-check
Retry / loop
Max 3 total attempts (initial + 2 retries). Each retry: build-error-cleanup agent fixes errors → build-check script re-runs. Exhausted → return failed with error details.
Returns
Status to run-process-change via build-result.json: passed | failed
Writes (internal)
build-check.jsonbuild-error-cleanup.jsonbuild-result.json
Key rule: build-check is a bash script, not an agent — deterministic, no AI judgment. Detects project toolchain and runs build, lint, typecheck, tests. Only build-error-cleanup uses an LLM.
File contract hierarchy
Each layer reads only its own level — never reaches into child orchestrator internals

run-build-check reads:
build-check.json · build-error-cleanup.json
